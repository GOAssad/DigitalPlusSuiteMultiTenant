# Inicio de sesion — 2026-03-04

## Que se hizo en esta sesion

### Objetivo cumplido: Azure Functions Provisioning Endpoint (doc 0010)

Se creo el proyecto completo de Azure Functions para que el instalador obtenga credenciales de BD sin que el usuario las vea, usando codigos de activacion de un solo uso.

### Decisiones de arquitectura

| Decision | Elegido | Motivo |
|---|---|---|
| Storage de ActivationCodes | BD Ferozo existente | $0 costo adicional |
| Secretos cloud | App Settings de Azure Functions | $0 (sin Key Vault) |
| Lenguaje | C# .NET 8 Isolated Worker | Consistente con el stack |
| Plan Azure Functions | Consumption (gratis) | 1M ejecuciones/mes gratis |

### Flujo del endpoint

```
Installer → POST /api/provision (con X-Api-Key)
  ├─ 400: payload invalido
  ├─ 401: API key incorrecta
  ├─ 403: codigo invalido / expirado / ya usado
  ├─ 409: BD ya existe en cloud
  ├─ 500: error creando BD
  └─ 200: { companySanitized, dbName, mode, connectionString, policy }
```

### Fase 3: Sistema de licencias y activacion — Version 1.3

Implementar un sistema de licencias para controlar el uso del software en funcion del pago.

**Modelo**: freemium — gratis hasta una cantidad de transacciones (fichadas) determinada, luego requiere licencia paga para seguir operando.

**Por definir antes de implementar** (el usuario debe decidir):
- Cuantas transacciones gratis (ej: 100? 500? 1000?)
- Periodo de la licencia paga (mensual? anual?)
- Precio / modelo de cobro
- Que pasa al vencer: bloqueo total o modo consulta?

**Componentes a crear**:
- [ ] **Tabla Licenses en Ferozo**: LicenseKey, CompanyName, DbName, MaxTransactions, ExpiresAt, IsActive, PlanType
- [ ] **Endpoint Azure Function**: `POST /api/license/check` — recibe dbName + machineId, retorna estado de licencia y transacciones restantes
- [ ] **Endpoint Azure Function**: `POST /api/license/activate` — activa una licencia con una key
- [ ] **Contador de transacciones**: el Fichador lleva cuenta de fichadas realizadas y consulta periodicamente el endpoint
- [ ] **Pantalla de estado en Fichador/Administrador**: licencia activa, por vencer, limite alcanzado, transacciones restantes
- [ ] **Degradacion elegante**: al superar el limite free, el sistema sigue mostrando datos existentes pero no permite nuevas fichadas, con mensaje claro de como activar
- [ ] **Herramienta admin**: generar/renovar licencias (manual con script al inicio, portal web despues)
- [ ] **Subir version del instalador a 1.3**

**Nota**: esta fase reutiliza la misma infraestructura Azure Functions + Ferozo de la Fase 1. Se agregan endpoints y tablas, no se crea infraestructura nueva.

## Regla de versionado del instalador

Cada cambio significativo en el instalador debe incrementar la version:

| Version | Cambio |
|---|---|
| 1.0 | Instalador unificado original |
| 1.1 | DPAPI hardening + ACL (sesion 2026-03-03) |
| **1.2** | Provisioning con codigos de activacion + eliminacion de campos SQL (proxima) |
| **1.3** | Sistema de licencias freemium (siguiente) |

La version se define en `setup.iss`:
```
#define AppVersion    "1.2"
```
Y se refleja en el nombre del output:
```
OutputBaseFilename=DigitalPlus_Suite_Setup_v{#AppVersion}
```
