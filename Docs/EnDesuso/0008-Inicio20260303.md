# Objetivo (Etapa 1): Evitar que el usuario vea user/pass de SQL en los .config y endurecer la seguridad básica

Tenemos 2 apps .NET Framework 4.8 (Fichador TEntradaSalida y Administrador) instaladas por Inno Setup.
Hoy, al instalarse, generan .exe.config con connection strings visibles en texto plano. Si el usuario abre el archivo, ve servidor/usuario/password.

Quiero que propongas y luego implementes (con cambios concretos y documentados) una solución REALISTA para escritorio .NET 4.8 que:
1) Evite que el usuario vea credenciales en texto plano.
2) No dependa de que el usuario sepa datos técnicos.
3) No rompa el funcionamiento offline y no agregue complejidad ridícula.

## Restricciones y hechos actuales
- Instalador unificado ya genera los .config desde templates.
- Queremos mantener la experiencia simple: instalar y usar.

## Lo que necesito que entregues en esta etapa
A) Un análisis corto con 3 alternativas, con pros/cons y recomendación:
   - Alternativa 1: propone la que encuentres que no sean la Alternativa 2 y la Alternativa 3.
   - Alternativa 2: Encriptar la sección connectionStrings del .config usando ProtectedConfiguration (DPAPI Machine-level) para .NET Framework.
   - Alternativa 3: Servicio local (Windows Service) o helper que guarde secretos en DPAPI/Windows Credential Manager y la app consuma por IPC (Named Pipes) sin exponerlos en config.
   Elegí una como recomendada para nuestro caso y explicá por qué.

B) Implementación mínima viable (MVP) de la recomendación elegida:
   - Si elegís DPAPI/ProtectedConfiguration:
     1) En el instalador: al final, encriptar sección `connectionStrings` (y si aplica `appSettings`) del config de ambas apps.
     2) Documentar el comando o método exacto (por ejemplo aspnet_regiis o API de ProtectedConfiguration) y cómo se ejecuta desde Inno Setup o PowerShell.
     3) Confirmar lectura exitosa en runtime (la app sigue conectando).
   - Si elegís Service/IPC:
     1) Definir el esquema (qué corre como servicio, qué guarda, cómo se recupera).
     2) MVP implementado y documentado.

C) Hardening adicional obligatorio (aplicable a cualquier alternativa):
   - Poner permisos NTFS restrictivos al directorio de instalación y a los *.config (usuarios estándar solo lectura si corresponde).
   - Evitar dejar logs con connection strings completas.
   - Validar que no exista fallback hardcodeado de cadena (si existe, eliminarlo o bloquearlo).

## Output que quiero
- Un checklist técnico + archivos tocados.
- Cambios concretos en: templates de config, código si hace falta, y setup.iss.
- Notas de pruebas: instalar en una PC limpia y validar que las 2 apps conectan OK.
